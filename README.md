# Я отходы — документация проекта

Демо-сайт системы агрегации заявок на переработку вторичных ресурсов.  
На примере **соломы** показано, как завод, мелкие поставщики (ИП) и крупный перевозчик работают через одну платформу.

Это **учебная демонстрация**, а не промышленная система: нет регистрации, паролей и реальных платежей. Все роли открыты в браузере, данные хранятся локально в SQLite.

## Зачем нужна такая система

Заводу нужно много вторичного сырья — например, соломы после уборки урожая. Но:

- у завода **нет своих машин** и людей, чтобы собрать солому по всей области;
- солому могут сдать **мелкие фермеры и ИП** — по 5–15 тонн каждый;
- везти каждую маленькую партию отдельно на завод **дорого и неудобно**.

Решение — **агрегация**:

1. Завод размещает **один большой заказ**.
2. ИП сдают солому на **локальные склады** поближе к себе.
3. **Крупный перевозчик** забирает накопленный объём с нескольких складов и везёт **одним маршрутом** на завод.

Платформа «Я отходы» показывает эту цепочку в простом веб-интерфейсе.

---

## Главная идея

Система связывает три стороны в одном процессе:

```
Завод (заказ)  →  ИП (мелкие поставки на склады)  →  Логист (крупная доставка на завод)
```

**Склады** — промежуточные точки. Они не отдельная «роль» в меню сайта, но важная часть логики: именно на них собирается сырьё перед отправкой на завод.

## Участники и роли

### 1. Завод-потребитель

Промышленное предприятие, которому нужна солома (или другое вторсырьё).

**Что указывает в заказе:**

| Поле | Смысл |
|------|--------|
| Название завода | Кто заказывает |
| Объём, тонн | Сколько соломы нужно |
| Срок поставки | К какой дате ждёт сырьё |
| Сырьё | В демо всегда «Солома» |
| Предоплата ИП, ₽ | Общая сумма аванса на сбор (для демо) |
| Цена ИП, ₽/т | Сколько платят за каждую тонну от мелкого поставщика |
| Награда логисту, ₽ | Сколько платят перевозчику за рейс на завод |
| Адрес завода | Куда везти итоговый груз |
| Координаты завода | Для построения маршрута на карте |
| Основной склад сбора | С какого склада логически «ведётся» заказ |

---

### 2. ИП-поставщик (мелкий поставщик)

Индивидуальный предприниматель или фермер, у которого есть небольшой объём соломы (обычно 5–12 тонн).

**Что делает ИП:**

1. Видит **активные заказы завода** и **адреса локальных складов**.
2. **Оформляет заявку** — сообщает, сколько тонн готов привезти и на какой склад.
3. **Физически доставляет** солому на склад.
4. **Закрывает поставку** в системе: указывает объём, прикрепляет фото, подтверждает геолокацию (GPS).

Заявка ИП и фактическая поставка — **два разных шага**. Сначала ИП «бронирует» объём, потом подтверждает, что реально сдал.

---

### 3. Крупный перевозчик (логист)

Компания или ИП с большим транспортом (фуры, трал).

**Что делает логист:**

1. Видит **заказы завода**, которые готовы к перевозке или скоро будут готовы.
2. Видит **локальные склады**, с которых нужно забрать солому.
3. Видит **маршрут**: несколько складов по порядку → финальная точка (завод).
4. Видит **награду за рейс**.
5. Берёт заказ в работу и отмечает **доставку на завод**.

В демо маршрут можно открыть в **Яндекс.Картах** — система собирает ссылку из координат всех точек.

## Как проходит заказ от начала до конца

Пошаговый сценарий на примере одного заказа:

```
Шаг 1. Завод размещает заказ
        «Нужно 45 т соломы до 15 числа, адрес — Муром»

Шаг 2. ИП видят заказ и склады
        ИП Сидоров: «Привезу 8 т на Тульский хаб»
        ИП Козлова: «Привезу 12 т на Рязанский узел»

Шаг 3. ИП доставляют солому на склады
        Каждый закрывает поставку: фото + GPS + объём

Шаг 4. Система считает собранный объём
        Когда на складах набралось ≥ 45 т → статус «на_складе»

Шаг 5. Логист видит маршрут
        Рязанский узел (14 т) → Тульский хаб (10 т) → Калужский терминал (8 т) → Завод

Шаг 6. Логист забирает груз
        Статус «в_дороге»

Шаг 7. Логист доставил на завод
        Статус «завершён», логист получает награду (в демо — просто отметка)
```

## Ключевые понятия

### Заказ завода (`factory_orders`)

Центральная сущность системы. Вокруг одного заказа собираются заявки ИП, поставки, маршрут логиста и статус выполнения.

### Заявка ИП (`supplier_offers`)

**Намерение** поставить определённый объём на конкретный склад по конкретному заказу.

- Статус **«активна»** — ИП обещал привезти, но ещё не закрыл поставку.
- Статус **«доставлено»** — ИП уже сдал солому и закрыл поставку в системе.

### Поставка (`supplier_deliveries`)

**Факт** сдачи соломы на склад. Содержит:

- имя ИП;
- объём в тоннах;
- координаты GPS;
- фото (необязательно, но можно загрузить);
- статус приёмки.

### Локальный склад (`warehouses`)

Промежуточная точка сбора. У каждого склада есть название, адрес и координаты на карте.

В демо три склада:

| Склад | Регион |
|-------|--------|
| Тульский хаб | Тульская область |
| Рязанский узел | Рязанская область |
| Калужский терминал | Калужская область |

### Маршрут логиста (`order_route_stops`)

Упорядоченный список складов, которые логист посещает **по пути** к заводу.

Каждая точка маршрута хранит:

- **порядковый номер** (`stop_order`) — 1, 2, 3…;
- **склад** — куда заехать;
- **тонн на складе** (`tons_at_stop`) — сколько соломы там лежит для этого заказа.

Последняя точка маршрута — всегда **завод** (его адрес хранится в заказе, не в таблице остановок).

### Каталог ИП (`supplier_profiles`)

Справочник мелких поставщиков: имя, регион, максимальный объём, который могут дать.  
На странице завода показан для обзора — «кто потенциально может участвовать в сборе».

### Сбор объёма

Система **суммирует все поставки** по заказу (со статусами «сдано» и «принято») и сравнивает с нужным объёмом заказа.

Формула простая:

```
собрано = сумма(quantity_tons) по всем поставкам заказа
если собрано ≥ quantity_tons заказа → можно переводить в статус «на_складе»
```

---

## Статусы заказа завода

| Статус | Что означает | Кто действует |
|--------|--------------|---------------|
| **сбор** | ИП ещё возят солому на склады, объём не набран | ИП |
| **на_складе** | Нужный объём собран, логист может забирать | Логист |
| **в_дороге** | Логист забрал груз и везёт на завод | Логист |
| **завершён** | Сырьё доставлено на завод, заказ закрыт | — |

Переход **сбор → на_складе** происходит **автоматически**, когда сумма поставок достигает объёма заказа.

Переходы **на_складе → в_дороге → завершён** делает логист кнопками на своей странице.

### Статусы поставки ИП

| Статус | Смысл |
|--------|--------|
| **сдано** | ИП закрыл поставку, склад ещё не подтвердил |
| **принято** | Склад принял (страница `/warehouse`, в меню не выведена) |

Обе стадии **учитываются** при подсчёте собранного объёма.

## Что может делать каждая роль

### Завод (`/factory`)

**Просмотр:**

- список своих заказов со статусами;
- сколько тонн уже собрано;
- маршрут по складам (если задан);
- каталог ИП и их доступные объёмы.

**Действия:**

- создать новый заказ на солому через форму.

---

### ИП (`/supplier`)

**Просмотр:**

- список локальных складов с адресами и ссылками на карту;
- активные заказы завода (статус «сбор»);
- свои и чужие заявки (демо);
- история последних поставок.

**Действия:**

1. **Подать заявку** — указать ИП, объём, заказ, склад.
2. **Закрыть поставку** — указать объём, загрузить фото, получить GPS через браузер, отправить форму.

> Геолокация работает только если браузер разрешит доступ к местоположению. На компьютере координаты будут приблизительными — для демо это нормально.

---

### Логист (`/logistics`)

**Просмотр:**

- все локальные склады;
- заказы в статусах «сбор», «на_складе», «в_дороге»;
- для каждого заказа: сырьё, срок, награда, адрес завода;
- **визуальный маршрут** через склады;
- ссылка «Построить маршрут на карте» (Яндекс);
- история завершённых рейсов.

**Действия:**

- **«Забрать со складов»** — когда статус «на_складе»;
- **«Доставлено на завод»** — когда статус «в_дороге».

## Структура проекта

```
ya-othody/
├── app.py              # Основное приложение: маршруты, логика, база данных
├── database.db         # SQLite-база (создаётся автоматически)
├── requirements.txt    # Зависимости Python
├── Procfile            # Команда запуска для Heroku/Render
├── render.yaml         # Настройки деплоя на Render.com
├── templates/          # HTML-шаблоны страниц
│   ├── base.html       # Общий каркас: шапка, меню, стили
│   ├── home.html       # Главная
│   ├── factory.html    # Завод
│   ├── supplier.html   # ИП
│   ├── logistics.html  # Логист
│   └── warehouse.html  # Склад (служебная страница)
└── static/
    └── uploads/        # Загруженные фото поставок
```

### Технологии (кратко)

| Компонент | Технология |
|-----------|------------|
| Сервер | Python, Flask |
| База данных | SQLite (файл `database.db`) |
| Интерфейс | HTML + Tailwind CSS (через CDN) |
| Продакшен-сервер | Gunicorn |

Нет отдельного фронтенда (React/Vue) — страницы рисует сервер из шаблонов. Для демо этого достаточно.

| Поле | Описание |
|------|----------|
| order_id | Заказ |
| supplier_name | ИП |

| warehouse_id | Склад |

## Запуск локально

### Требования

- Python 3.12+ (или 3.14 — как в вашем venv)
- pip

### Установка и запуск

```bash
cd ya-othody

# Создать виртуальное окружение (один раз)
python3 -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate

# Установить зависимости
pip install -r requirements.txt

# Запустить сервер
python app.py
```

Откройте в браузере: **http://localhost:5001**

### Переменные окружения

| Переменная | По умолчанию | Назначение |
|------------|--------------|------------|
| `PORT` | `5001` | Порт сервера |
| `FLASK_DEBUG` | `1` | Режим отладки (автоперезагрузка) |
| `SECRET_KEY` | demo-ключ | Секрет Flask для сессий |

## Публикация в интернет

Проект готов к деплою на **Render.com** (файл `render.yaml`) или **Heroku** (файл `Procfile`).

Команда запуска в продакшене:

```bash
gunicorn app:app --bind 0.0.0.0:$PORT
```

> На бесплатном хостинге файл `database.db` может **сбрасываться** при перезапуске. Демо-данные создадутся заново при старте приложения.

## Ограничения демо-версии

Это **прототип для показа идеи**, а не готовый продукт. Важно понимать, чего здесь **нет**:

| Нет в демо | Что было бы в реальной системе |
|------------|--------------------------------|
| Регистрация и вход | Личные кабинеты с паролями |
| Разделение прав | Каждый видит только свои заказы |
| Реальные платежи | Эквайринг, договоры, акты |
| Проверка фото и GPS | Антифрод, сверка с адресом склада |
| Уведомления | SMS, email, push |
| Мобильное приложение | Отдельное приложение для ИП в поле |
| Выбор логиста | Тендеры, аукционы ставок |
| Другие виды сырья | В демо только солома |

Демо-данные **пересоздаются** при изменении константы `DEMO_SEED_VERSION` в `app.py` (сейчас версия `2`). Пользовательские заказы при этом удаляются — это ожидаемое поведение для учебного стенда.

## Краткий глоссарий

| Термин | Простое объяснение |
|--------|-------------------|
| **Вторсырьё** | Материал для переработки: солома, опилки, макулатура и т.п. |
| **Агрегация** | Сбор многих мелких партий в один крупный поток |
| **Локальный склад** | Ближайшая точка, куда ИП может сдать солому без поездки на завод |
| **Заявка ИП** | Обещание привезти N тонн |
| **Поставка** | Подтверждение, что N тонн уже сданы |
| **Маршрут** | Цепочка складов + завод для крупного перевозчика |
| **Награда** | Оплата участнику (в демо — просто число в интерфейсе) |

## Контакты и развитие

Проект создан как наглядная модель цепочки **завод → ИП → склад → логист** для переработки вторичных ресурсов.

Возможные направления развития (не реализованы):

- авторизация по ролям;
- несколько видов сырья;
- автоматический подбор маршрута;
- интеграция с картами и трекингом транспорта;
- финансовый модуль и отчёты.
